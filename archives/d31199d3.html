鸟瞰MySQL全貌

MySQL首先分为服务层与存储引擎层，服务层会调用存储引擎层的接口，存储数据或者查询数据。其中，服务层从外到内可以分为连接器、分析器、优化器、执行器。

连接器

连接器的任务是接收客户端发起的连接的，连接建立后如果一直没有操作，则连接会处于Sleep状态、即空闲状态，如果空闲状态持续了8小时，则连接器会断开次连接。连接可以分为长连接和短连接，长连接是指如果客户端持续有请求，则一直使用同一个连接，短连接是指，执行几条SQL之后就会断开连接，建立新的连接。一个长连接在执行大量的SQL之后，会占用临时内存，只有连接断开后，内存才会释放，解决此问题的方法有两个，定期断开长连接，初始化连接状态。

查询缓存

查询缓存会以key-value的形式存储在内存中，求中key是SQL语句，value是查询的结果。查询时如果命中缓存则不用进行磁盘IO，如果没有命中，则先进行磁盘IO查询数据，在存入内存中。但一旦对表进行修改后，次表的查询缓存就要失效，所以查询缓存的失效非常频繁，故，从MySQL 8.0开始，不再支持查询缓存的功能了。

分析器

分析器会先做词法分析、再做语法分析。词法分析会识别每一个字符，语法分析会分析SQL是否符合语法。

优化器

优化器是决定如何执行SQL的，比如，决定使用哪个索引，决定连接查询时表的连接顺序。

执行器

执行器回去调用存储引擎的接口，对数据进行操作。

日志系统

redo log

将对数据的修改直接写到磁盘中是随机IO，比较耗时，为了解决这个问题，有了redo log。

将修改先写到redo log中，然后在空闲时，再写到磁盘中。虽然redo log也在磁盘中，但是redo log是顺序写，所以速度更快。redo log是循环写，即redo log的空间是有限的，先慢之后，会擦除之前的日志。

redo log是innoDB提供的。

bin log

bin log是用来记录对数据库的逻辑操作，配合全量备份来恢复数据库。

bin log是追加写，在外存中。

‘bin log是MySQL服务层提供的。

两阶段提交模型

两阶段提是为了维护，两个日志的逻辑一致性。防止用bin log + 全量备份恢复出来的数据库和数据库不一致。

redo log 等待 binlog 写入完成后，由 prepare 变为 commit 提交状态

事务隔离

隔离级别

读未提交：一个事务还没有提交时，它做的变更就能被别的事务看到。直接返回记录上的最新值，没有视图概念。

读提交：一个事务提交之后，它做的变更才能被别的事务看到。每个SQL语句开始执行的时候创建视图。

可重复读，事务执行期间看到的数据是前后一致的。事务启动的时候创建视图。

串行化，读加读锁，写加写锁。

这里的视图并不是指用查询语句定义的虚拟表，而是InnoDB在实现MVCC时用的一致性读视图，即consistent read view

索引

在InnoDB中，一张表对应一个主键索引和0~多个非主键索引。

主键索引的叶子结点存储的是整行数据，又叫聚集索引。非主键索引的叶子结点存储的是主键的值，又叫非聚集索引、二级索引。

自增主键的好处

每次增加一条新记录都是追加操作，如果之前的页满了，直接申请一页就好了，不用移动之前的记录。如果不是自增的，可能会触发页分裂，移动之前的行记录。

回表查询与覆盖索引

回表查询是指，先查二级索引查出主键的值之后，再查聚集索引，查出行记录。

覆盖索引：为了避免回表查询，如果二级索引就已经包含了所需要的数据，则不需要进行回表查询，所以，覆盖掉需要查询的数据的索引就是覆盖索引。

最左前缀原则

索引项是按照索引定义里面出现的字段顺序排序的，所以查询的时候，也要符合这个顺序才能查询。

索引下推

在索引遍历过程中，对索引包含的字段先做判断，直接过滤掉不满足条件的记录，减少回表次数。

全局锁

全局锁是对整个数据库实例加锁。在对全库做逻辑备份时会用到。

MySQL 提供了一个加全局读锁的方法，命令是 Flush tables with read lock (FTWRL)，执行次命令后，数据库会处于只读状态。

但是为了备份让全库处于只读状态会有影响：

  • 如果主库处于只读，则业务基本停摆。
  • 如果从库处于只读，则从库无法执行从主库同步过来的bin log，导致主从延迟。

如果所有表均支持事务，则使用逻辑备份工具mysqldump时，使用参数–single-transaction

为何不用set global readonly=true的方式让全库只读？

在有些系统中，readonly 的值会被用来做其他逻辑，比如用来判断一个库是主库还是备库。

如果执行 FTWRL 命令之后由于客户端发生异常断开，那么 MySQL 会自动释放这个全局锁，整个库回到可以正常更新的状态。而将整个库设置为 readonly 之后，如果客户端发生异常，则数据库就会一直保持 readonly 状态，这样会导致整个库长时间处于不可写状态，风险较高。

表级锁

表锁

通过 lock tables … read/write给表加读锁或写锁。

元数据锁（MDL）

不用显式使用，当对一个表做增删查改操作时，加MDL读锁。当对一个表做修改表结构操作时，加MDL写锁。

行锁

行锁是由存储引擎实现的。

两阶段锁协议

在InnoDB事务中，行锁是在需要的时候加上的，但并不是不需要的时候就可以立即释放，要等到事务结束时，才能释放锁。也就是说，越晚加锁，则锁的持续时间越短，越有利于并发。

死锁和死锁检测

不同的线程出现循环资源依赖，涉及的线程都在等待别的线程释放资源，这些线程就进入了死锁状态。

死锁后的解决办法：

  • 等待到超时
  • 发起死锁检测，主动回滚某个事务。

一致性读视图

在可重复度隔离级别下，事务在启动的时候就获取了一个一致性读视图，这个一致性读视图是基于整库的。

InnoDB 里面每个事务有一个唯一的事务 ID，叫作 transaction id。它是在事务开始的时候向 InnoDB 的事务系统申请的，是按申请顺序严格递增的。而每行数据也都是有多个版本的。每次事务更新数据的时候，都会生成一个新的数据版本，并且把 transaction id 赋值给这个数据版本的事务 ID，记为 row trx_id。同时，旧的数据版本要保留，并且在新的数据版本中，能够有信息可以直接拿到它。旧版本的数据并不是物理上真实存在的，而是新版本的数据借助undo log计算出来的。

事务在启动时，会将此刻已经启动但还未提交的事务的ID存储到一个数组中，数组中事务ID的最小值成为低水位，当前系统中已经创建的事务ID的最大值+1称为高水位。

高低水位就将系统中的事务分为了3种，已提交事务、未提交事务、未开始事务。

如果一行数据的事务ID小于低水位，则此行数据是可见的。

如果一行数据的事务ID位于低水位和高水位之间，则判断此ID是否在数组中，如果在，则可见，如果不在，则不可见。

如果一行数据的事务ID大于等于高水位，则数据不可见。

更新数据时，是先读后写，而这个读，是当前读，不会根据undo log会到之前版本的数据。

普通索引和唯一索引

查询过程

走普通索引查到第一个满足条件的记录后，还需要查找下一条记录。走唯一索引查找到第一个满足条件的记录后，不许查找下一条记录。但是因为磁盘IO是以数据页为单位的，所以，基本上下一条记录就在内存中了，但不排除下一跳正好在下一页的情况。

更新过程

Buffer Pool

缓冲池缓存磁盘上经常操作的真实数据，在执行增删改查操作时，先操作缓冲池中的数据（未命中，则从磁盘加载并缓存，然后再以一定的频率刷新到磁盘，从而减少IO。

Change Buffer

更改缓冲区，针对非唯一二级索引页，在执行DML语句时，如果数据页不在Buffer Pool中，不会直接进行磁盘IO，而是将数据变更存在更改缓冲区Change Buffer中，在未来数据被读取时，再将数据合并到Buffer Pool中，再将合并后的数据刷新到磁盘中。

对于唯一索引来说，更新操作要判断操作是否违反唯一性约束，需要先将数据页读入Buffer Pool中，既然已经在Buffer Pool中了，那就没必要再用change buffer了，因为用了change buffer后，还需要merge到buffer pool中，多此一举了。

如果更新的记录所在的目标页不在buffer pool中，对普通索引来说，直接更新在change buffer中即可，对于唯一索引来说，需要将数据读入buffer pool中，判断是否违反唯一性约束， 在修改buffer pool中的内容。

Change Buffer的使用场景

如果写多读少，则将修改累计在Change Buffer中，一直到读的时候，才会进行磁盘IO。这样就减少了磁盘IO次数。

如果写少读多，则将修改累计在Change Buffer后，马上就要进行磁盘IO了，而且比不用Change Buffer的情况还多了merge的操作，反而降低了效率。

优化器的逻辑

优化器选择索引的依据：扫描行数、是否使用临时表、是否排序。

扫描行数如何判断

执行语句之前并不能知道会扫描多少行，所以用“基数”来估算行数。基数：一个索引上不同的值的个数，称之为基数。

如果一行一行的取出来计算基数代价太高，所以选择“采样统计”。采样统计：统计N个数据页上的平均不同值，然后乘以总页数。当超过一定数量（总行数的1/M）的数据行发生了变化后，会重新做一次统计。

在 MySQL 中，有两种存储索引统计的方式，可以通过设置参数 innodb_stats_persistent 的值来选择：设置为 on 的时候，表示统计信息会持久化存储。这时，默认的 N 是 20，M 是 10。设置为 off 的时候，表示统计信息只存储在内存中。这时，默认的 N 是 8，M 是 16。
